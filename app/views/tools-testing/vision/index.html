Understand the impact

Being able to use sight to interact with or understand content can be affected by a range of conditions, including:

  • colour blindness (difficulty distinguishing certain hues or contrasts; see WebAIM: Understanding WCAG contrast requirements)
  • low vision (reduced acuity requiring larger text, higher contrast, magnification; see GDS Persona: Claudia, partially sighted magnifier user)
  • blindness (little or no usable vision, relying on screen readers or other assistive technologies; see GDS Persona: Ashleigh, screen reader user)
  • eye injuries or conditions (e.g., cataracts, macular degeneration, glaucoma) leading to variable vision capabilities; refer to general WCAG context (WCAG Perceivable Principle)

People with visual impairments may:

  • have difficulty reading small or low-contrast text, especially below a 4.5:1 contrast ratio (see WCAG 2.2 SC 1.4.3 Contrast (Minimum) and WebAIM contrast guidance)
  • need to resize text up to 200% without loss of content or functionality (see W3C Understanding SC 1.4.4 Resize Text)
  • experience layout breakage when zooming or adjusting spacing (see WCAG 2.2 SC 1.4.12 Text Spacing and WebAIM: Text Spacing Guidance)
  • struggle to distinguish information conveyed by colour alone (see WebAIM: Evaluating Contrast and Color Use)
  • require clear focus indicators and sufficient target sizes to navigate via keyboard or magnifier (see MDN: Forced Colors Mode and focus styles)
  • benefit from screen reader support (for partially sighted or blind users) with correct semantic markup and ARIA roles (see WebAIM: Designing for Screen Reader Compatibility and W3C WAI-ARIA Overview)

To illustrate real-world experiences, consider GOV.UK accessibility personas:

  • Claudia – a partially sighted magnifier user who relies on enlarged text and high contrast to read content (GOV.UK: Claudia persona profile)
  • Ashleigh – a severely sight impaired screen reader user who depends on correct semantic structure and ARIA support to navigate pages without visual cues (GOV.UK: Ashleigh persona profile)

Understanding these personas helps teams empathise and simulate tests that reveal barriers (see GOV.UK Blog: Using persona profiles to test accessibility).

Tools you can use

There are a range of tools and techniques you can use to test for visual impairments, organised here by purpose with plain-language guidance.

Colour contrast checker

The colour contrast checker measures whether text (or images of text) and background colours meet WCAG 2.2 contrast requirements (minimum 4.5:1 for normal text, 3:1 for large text; see WCAG 2.2 SC 1.4.3 Contrast (Minimum) and WebAIM contrast guidance).

How to use: Use tools such as WebAIM Contrast Checker or Colour Contrast Analyser to input or pick foreground and background colours and verify ratios. In browser devtools, inspect computed styles for dynamic content or overlayed text.

Interpretation: If the ratio is below threshold, revise design: increase font weight, adjust colours, add background overlays, or change layout. Automated results cover only some contexts; always review in context (e.g., text over images; see Deque University: color-contrast rule).

Visual simulation scripts (formerly bookmarklets)

These browser-based simulation scripts let you approximate low-vision experiences by applying effects like blur, grayscale, or magnification. Although technically implemented as “bookmarklets,” refer to them as visual simulation scripts for clarity (see GDS Accessibility personas).

How to install: Drag the provided link to your favourites bar or add the snippet as a browser extension/user script. Label descriptively (e.g., “Blur simulation”, “Grayscale simulation”).

How to test: Activate the script on a page to simulate blurred or desaturated vision and observe which elements become unreadable or confusing. Use zoom scripts to enlarge content beyond 200% and check layout resilience; use spacing adjustment scripts to test text spacing requirements (see WebAIM: Text Spacing Guidance).

Interpretation: Identify failures (e.g., text too small or low contrast under blur), then adjust design: larger fonts, higher contrast, flexible layouts using relative units. Document issues with reference to personas (e.g., Claudia’s difficulty reading blurred headings).

Screen reader testing

For partially sighted users who supplement vision with audio, and blind users relying entirely on audio, test with popular screen readers (e.g., NVDA on Windows, VoiceOver on macOS/iOS; see WebAIM: Screen Reader Compatibility).

How to use: Enable the screen reader, navigate via headings, landmarks, links, and form controls. Verify announcements are meaningful, labels present, focus order logical, and dynamic content announced via ARIA live regions (see W3C WAI-ARIA Overview and WebAIM: Testing with Screen Readers).

Interpretation: If content is skipped, misannounced, or unlabeled, fix semantic markup (proper HTML elements, ARIA attributes). Document scenarios with reference to persona Ashleigh’s experience (e.g., difficulty finding “Submit” button due to missing accessible name).

Zoom and resize testing

Ensure text can be resized up to 200% without loss of content or functionality, per WCAG 2.1 SC 1.4.4 (Understanding Resize Text).

How to test: In browser settings or via simulation scripts, increase text size or page zoom to 200% and navigate all pages; check that content remains visible, no horizontal scrolling is required, and interactive elements remain operable.

Interpretation: If layout breaks (overlaps, off-screen content), refactor CSS to use relative units (em, rem, %) and flexible containers (flexbox/grid). Test with persona Claudia’s profile (magnifier + high zoom; see GOV.UK Blog: Remote accessibility persona testing).

Text spacing testing

Verify that increasing line height, letter spacing, word spacing, and paragraph spacing does not cause loss of content or functionality, per WCAG 2.2 SC 1.4.12 (WCAG Text Spacing and WebAIM: Text Spacing Guidance).

How to test: Use simulation scripts or browser extensions that apply spacing adjustments; review pages for overlap or cutoff. Combine with zoom testing to simulate compounded effects.

Interpretation: If text overlaps or is clipped, adjust CSS to avoid fixed heights, use min-height, flexible line boxes, and avoid positioning that breaks under spacing changes.

Target size highlighter

Identify interactive elements smaller than recommended size (at least 24x24 CSS pixels) to ensure ease of activation when zoomed or via magnifier (see WebAIM: Evaluating Contrast and Color Use for guidelines on visibility under zoom).

How to use: Activate a script or use devtools to outline clickable areas; measure dimensions. Many simulation scripts include target-size highlighting.

Interpretation: Increase padding or redesign controls to meet minimum size; test under zoom and with magnifier to simulate persona Claudia’s needs.

Developer / browser devtools techniques

Use browser devtools to inspect CSS for colour values, simulate forced-colors mode, emulate vision deficiencies (see MDN: Forced Colors Mode), and test responsive behaviour under zoom.

How to test: In Chrome DevTools, under “Rendering” or via extensions, apply emulated vision deficiencies; inspect computed styles for contrast; simulate high-contrast mode; check focus styles under forced-colors.

Interpretation: Address issues uncovered: ensure focus indicators are visible (contrast ≥3:1 to adjacent pixels), semantic HTML, ARIA where needed; validate under real assistive tech as final check.

Scenario: Improving readability for low-vision users

Claudia uses a magnifier at 150% and blur simulation; she notices low contrast on form labels and overlaps when zoomed; the team updates colour palette for higher contrast, refactors CSS to flex layout, and verifies via screen reader that labels remain associated when resized (see WebAIM: Testing with Screen Readers and GOV.UK Blog: Using persona profiles to test accessibility).

Who should test

  • Designers: choose accessible colour palettes, typography, spacing, focus indicators (see GDS Design Principle: Start with user needs).
  • Developers: implement relative units, semantic HTML, ARIA support; integrate automated contrast checks in CI/CD pipelines using tools like axe DevTools (see Deque University: color-contrast rule).
  • Content writers: ensure clear headings, sufficient text size, avoid conveying info by colour alone (see WebAIM contrast guidance).
  • QA/Testers: include zoom, contrast, screen reader tests in regression suites; simulate personas via profiles or scripts (see GOV.UK Blog: Remote accessibility persona testing).
  • Product owners: prioritise fixes based on impact to personas like Claudia and Ashleigh; allocate time/budget for thorough visual accessibility testing (see GOV.UK User Profiles).
